# 版本迭代规范

## 目标

统一项目的版本开发、文档归档和发布节奏，避免版本说明散落在根目录。

## 版本号规则

使用 `主版本.次版本.修订号`：

- `X.0.0`：大阶段重构、架构切换、兼容性变化
- `1.X.0`：新增功能、模块扩展、明显体验升级
- `1.0.X`：Bug 修复、稳定性优化、小范围体验修正

Flutter 代码版本以 `flutter-app/pubspec.yaml` 为准。
打包版本以 `build.bat` 中版本号为准。
两处版本号在发版前必须同步。

## 文档归档规则

每个版本统一放在 `versions/vX.Y.Z/`，最少维护 4 个文件：

- `README.md`
- `瞬-vX.Y.Z-版本说明.md`
- `瞬-vX.Y.Z-Bug修复记录.md`
- `瞬-vX.Y.Z-测试报告.md`

允许补充：

- `测试清单.md`
- `性能优化报告.md`
- 其他该版本专属文档

禁止再把 `V1.0.X-xxx.md` 直接放在仓库根目录。

## 每次迭代流程

1. 确认本次版本号，例如 `v1.0.4`。
2. 修改 `flutter-app/pubspec.yaml` 与 `build.bat`。
3. 新建 `versions/v1.0.4/`，按上一版本最小文档集创建版本文档。
4. 完成开发与联调。
5. 更新 `versions/CHANGELOG.md`。
6. 更新 `versions/版本索引.md`。
7. 执行本地测试和发布前检查。
8. 确认 `git status` 无异常后，由你手动 push 到远程仓库。

## 建议的发版检查

- `flutter test`
- 必要时 `flutter analyze`
- 后端关键测试
- 真机登录、匹配、聊天、图片上传、设置同步回归
- 文档中的版本号、日期、状态一致

## Git 协作约定

- 本仓库继续连接远程 Git 仓库。
- 每次版本迭代完成后，你手动执行 `git push`。
- 推送前建议至少完成：
  - 版本号更新
  - `versions/` 文档更新
  - 最小回归验证

- 版本性文档全部归档到 `versions/`
- `versions/v1.0.4/` 作为当前最新版本文档入口
